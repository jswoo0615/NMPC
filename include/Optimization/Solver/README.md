# Solver Module

This directory contains the numerical optimization core of the NMPC system. It is responsible for efficiently solving the large, sparse Karush-Kuhn-Tucker (KKT) linear systems generated by the Sequential Quadratic Programming (SQP) and Interior Point Method (IPM) algorithms. The entire module is designed around hard real-time execution bounds, zero dynamic memory allocation, and ultimate autonomous driving safety.

## Core Components

### 1. `RiccatiSolver.hpp`
A high-performance discrete-time Riccati Recursion solver tailored for NMPC.
- **$O(N)$ Complexity**: Solves the block-tridiagonal KKT system generated across the prediction horizon in linear time ($O(N)$), drastically outperforming generic dense solvers that scale at $O(N^3)$.
- **Zero-Allocation & Zero-Copy**: Exclusively utilizes `matrix::StaticMatrix` and compile-time `std::array`. During the heavy Backward Pass, it directly utilizes hardware-accelerated `linalg::multiply_AT_B` to calculate transposed matrix products, completely avoiding the latency penalty of allocating and copying transposed matrices.
- **Levenberg-Marquardt Regularization**: Natively supports State and Control Hessian regularization (`reg_x`, `reg_u`), a critical mathematical feature used to stabilize the SQP solver when the KKT matrix temporarily loses positive-definiteness during highly non-linear maneuvers.

### 2. `MeritLineSearch.hpp`
Implements an Armijo Backtracking Line Search to determine the optimal step size ($\alpha$) during each SQP Newton step.
- Prevents "overshooting" in highly non-linear regions, guaranteeing global mathematical convergence.
- **Micro-Optimization**: Employs extreme compiler-level optimizations such as loop hoisting (extracting loop-invariant branches) and strength reduction (replacing repeated complex multiplications with recursive scalar scaling) to squeeze every possible microsecond out of the tightest loop in the NMPC algorithm.

### 3. `KKTMonitor.hpp`
A standalone, ultra-fast engine that evaluates the exact KKT mathematical optimality conditions (Stationarity, Primal Feasibility, Dual Feasibility, and Complementary Slackness).
- **Extreme SIMD Optimization**: Completely bypasses C++ standard library paradigms (like `std::initializer_list` or `std::vector`) to avoid touching stack or heap memory. It forces the evaluation of Infinity Norms ($L_\infty$) entirely within **ARM NEON** or **AVX2** CPU registers, providing instantaneous solver health checks without latency overhead.

### 4. `FallbackControl.hpp` & `SolverStatus.hpp`
The ultimate safety net linking abstract optimization mathematics to physical vehicle safety.
- **`SolverStatus`**: Explicitly categorizes the numerical health of the solver into safe (e.g., `SUCCESS`), degraded (e.g., `MAX_ITERATION_REACHED`), or critical (e.g., `MATH_ERROR`, `INFEASIBLE`).
- **`FallbackControl`**: Actively monitors the magnitudes of dual variables ($\mu$) and barrier slack variables ($s$). If it detects dual variable explosion (indicating mathematically conflicting constraints) or massive slack violations (indicating an unavoidable physical collision), it immediately overrides the NMPC output and injects a deterministic safe command (e.g., maintain straight steering and apply maximum emergency deceleration).
